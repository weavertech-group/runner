# T3 Code session

The private runner workflow can optionally start T3 Code for the selected target
repository and expose it through a pre-created Cloudflare Named Tunnel. The
hostname is stable across runs, while the runner, T3 state, and pairing
credential remain ephemeral.

See also the repository-wide [security policy and threat model](../SECURITY.md).

## Cloudflare setup

Create one remotely-managed Named Tunnel for each opaque target. Configure a
separate public hostname for each service:

```text
t3-repo-07.example.com  -> http://127.0.0.1:3773
mcp-repo-07.example.com -> http://127.0.0.1:7676
```

T3 Code and DevSpace must not share one hostname. Both services use root-level
HTTP, WebSocket, OAuth, and asset paths, so path-prefix routing is not supported.

The workflow only runs an existing Tunnel connector. It does not receive a
Cloudflare API token and does not create or delete Tunnels, DNS records, or
published application routes.

## GitHub Environment secrets

Add these secrets to the target's `session--<opaque-id>` Environment:

| Secret | Required when | Value |
| --- | --- | --- |
| `CLOUDFLARE_TUNNEL_TOKEN` | T3 Code or DevSpace is enabled | Token for the target-specific remotely-managed Tunnel |
| `T3_PUBLIC_URL` | `enable_t3code=true` | Root HTTPS origin, for example `https://t3-repo-07.example.com` |
| `DEVSPACE_PUBLIC_URL` | `enable_devspace=true` | Root HTTPS origin, for example `https://mcp-repo-07.example.com` |

Keep the Tunnel token in the Environment, not at repository scope. The runner
writes it to a mode-`0600` temporary file and passes that file to `cloudflared`.
The token is removed during finalization.

Do not configure two opaque targets with the same Tunnel token or hostname.
Workflow concurrency serializes sessions for the same opaque target, but it
cannot detect accidental credential reuse between different targets.

## Start a session

Dispatch **Private Runner Session** with:

- a non-empty allowed `target_id`;
- `enable_ssh=true`;
- `enable_t3code=true`;
- optionally `enable_devspace=true`.

The workflow then:

1. configures the target-scoped Git credential;
2. clones the selected repository once into a shared temporary workspace;
3. installs pinned `t3@0.0.28`;
4. adds the workspace as a T3 project;
5. starts T3 Code on `127.0.0.1:3773`;
6. starts DevSpace on `127.0.0.1:7676` when requested;
7. connects the pre-created Named Tunnel using its token file;
8. verifies public HTTPS access and the T3 WebSocket upgrade; and
9. stores connection information in private local files.

T3 Code and DevSpace operate on the same Git working tree.

## Read the pairing URL

Connect over the private tailnet and read:

```bash
tailscale ssh runner@gha-<run-id>-<run-attempt>
cat ~/private-runner-session/t3code/connection.txt
```

The file contains:

```text
T3_URL=https://t3-repo-07.example.com
PAIRING_URL=https://t3-repo-07.example.com/pair#token=<one-time-credential>
```

The fixed hostname is not printed to the public Actions log. The pairing
credential is generated by T3 Code for the current ephemeral runner and changes
on every workflow run.

Open `PAIRING_URL` in a browser or use it to add the environment in a T3 Code
client. Directly opening `T3_URL` is also supported after the environment has
been paired.

## State and persistence

The following are intentionally ephemeral:

- the cloned working tree;
- T3 project and thread state;
- T3 pairing and session credentials;
- DevSpace state and Owner Token;
- the active `cloudflared` connector.

The Named Tunnel, DNS record, and public hostname remain configured in
Cloudflare. When no runner is active, the stable hostname has no healthy origin.
Commit and push any required repository changes before the GitHub-hosted job
ends.

Do not place the T3 data directory, provider credential directories, private
source tree, or connection files in an Actions cache or artifact.

## Provider availability

The workflow installs current Codex CLI and Claude Code releases before starting
T3 Code. Installation does not by itself authenticate either provider. Provider
credentials must be supplied through a separately reviewed, non-interactive,
target-scoped mechanism before the corresponding provider can be used.

The first implementation does not install Grok, Cursor Agent, or OpenCode.

## Cloudflare Access

Cloudflare Access is not required by the workflow. T3 Code retains its own
pairing and session authentication.

Adding Access can affect browser redirects, cross-origin clients, and WebSocket
upgrades. Validate those clients separately before enabling an Access policy on
the T3 hostname. Do not treat an unlisted hostname as an authentication control.

## Failure codes

| Code | Meaning |
| --- | --- |
| `E50` | An optional public service was requested without a target or private SSH |
| `E51` | cloudflared installation, token/configuration, or connector startup failed |
| `E53` | The selected target repository could not be cloned |
| `E61` | The pinned T3 Code package could not be installed or verified |
| `E62` | T3 project initialization, startup, or local readiness failed |
| `E63` | Public T3 HTTPS or WebSocket readiness failed |

Detailed output remains under `$RUNNER_TEMP/private-runner-diagnostics` and is
not uploaded.
